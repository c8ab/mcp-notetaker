== Overview

MCP Notetaker is a Model Context Protocol (MCP) server that provides tools for recording, retrieving, and managing atomic notes. It is designed to run locally alongside AI-powered development tools (Claude Desktop, VS Code with Copilot, CLI agents) and persist notes in a SQLite database on the user's machine.

The system is a single-process Node.js application that communicates with MCP clients over stdio using the JSON-RPC-based MCP protocol. It exposes six tools for CRUD operations and full-text search on notes.

=== MCP Protocol Layer

The `McpServer` class from `@modelcontextprotocol/sdk` handles:

* MCP protocol negotiation (`initialize` handshake)
* Tool capability advertisement (`tools/list`)
* Request routing (`tools/call` dispatches to registered handlers)
* stdio transport (JSON-RPC over stdin/stdout)

Configuration is minimal. The entry point (`src/index.ts`) creates the server, registers all six tools, and connects to the stdio transport.

=== Tool Handlers

Six handler modules in `src/tools/`, one per MCP tool. Each handler:

1. Receives validated input (Zod schema validation is handled by the SDK)
2. Calls the appropriate NoteRepository method
3. Catches typed errors (`ParentNoteNotFoundError`, `NoteAlreadyDeletedError`) and returns `isError: true` MCP results
4. Returns JSON-serialized note records in `content[].text`

Handlers are deliberately thin -- they contain no business logic beyond input mapping and error formatting.

=== NoteRepository

A single class (`src/repository/note-repository.ts`) encapsulating all database operations:

* **Schema initialization:** Idempotent `CREATE TABLE IF NOT EXISTS` for the notes table, 6 indexes, and the FTS5 virtual table. WAL mode and foreign keys enabled on connection.
* **Prepared statements:** Frequently-used queries (insert, getById, softDelete, FTS operations) are prepared once on construction.
* **UUID v7 generation:** Custom implementation generating RFC 9562 UUID v7 identifiers with 48-bit millisecond timestamps for time-sortable primary keys.
* **FTS5 sync:** The `notes_fts` contentless FTS5 table is kept in sync at the application level -- rows are inserted on create, delete+re-insert on update, and deleted on soft-delete.
* **Tag querying:** Uses `json_each()` with AND semantics for tag filtering.
* **Transactions:** Create, update, and soft-delete operations use SQLite transactions to ensure atomicity between the notes table and FTS5 table.

== Data Architecture

A single SQLite database file stores all data. Default location: `~/.local/share/mcp-notetaker/notes.db` (configurable via `NOTETAKER_DB_PATH` environment variable).

=== Notes table

The primary table stores all note records with 12 columns:

* **Identity:** `note_id` (UUID v7, primary key)
* **Attribution:** `session_id`, `user_id`, `agent` (immutable after creation)
* **Content:** `title` (optional), `content` (AsciiDoc, required)
* **Relationships:** `parent_note_id` (self-referential FK for threading)
* **Metadata:** `context_url`, `tags` (JSON array in TEXT column)
* **Timestamps:** `created_at`, `updated_at`, `deleted_at` (ISO 8601)

Six indexes support the primary query patterns: filtering by session, user, agent, parent, creation time, and deletion status.

=== FTS5 virtual table

`notes_fts` is a contentless FTS5 virtual table indexing `title` and `content` columns. It uses the `unicode61` tokenizer for case-insensitive matching. The table is synced by the application layer (not triggers) to ensure soft-deleted notes are excluded from the search index.
